# ADR-0004: Token-Based Authentication

## Status

Accepted

## Context

DevDeck exposes a full shell via WebSocket at `/api/terminal` and serves an HTTP UI — both completely unauthenticated. Anyone who can reach the port gets unrestricted terminal access. Because DevDeck is a single-tenant, local-first developer tool (not a multi-user SaaS), a lightweight bearer-token scheme is sufficient to prevent unauthorized access without introducing user management overhead.

## Decision

1. **Token generation** — On startup the terminal server reads `DEVDECK_TOKEN` from the environment. If the variable is unset, a random token is generated via `crypto.randomUUID()`. The token and a ready-to-click URL are printed to stdout.

2. **Token storage** — A shared module `src/lib/auth.ts` exports `getToken()` (returns the current token) and `validateToken(candidate)` (constant-time comparison via `crypto.timingSafeEqual`).

3. **WebSocket protection** — The terminal server validates the token from the `token` query parameter on the HTTP upgrade request *before* spawning a PTY. Rejected connections receive WebSocket close code **4401** with reason `"Unauthorized"`.

4. **HTTP protection** — A Next.js middleware (`src/middleware.ts`) checks for a bearer token (`Authorization: Bearer <token>`) or a `token` query parameter on every request except static assets (`_next/static`, `favicon.ico`). On first visit with a valid `?token=` query param, the middleware sets a `devdeck_token` HTTP-only cookie and redirects to strip the token from the URL. Subsequent requests authenticate via the cookie.

5. **Frontend WebSocket auth** — `use-terminal.ts` appends `?token=<token>` to the WebSocket URL. The token value is read from the cookie (server-rendered pages) or from `localStorage` (client fallback). The hook also detects close code 4401 and surfaces an "Unauthorized" error state instead of reconnecting.

6. **Constant-time comparison** — All token comparisons use `crypto.timingSafeEqual` with UTF-8 encoded buffers to prevent timing side-channel attacks.

## Alternatives

| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| No auth (status quo) | Zero complexity | Full shell open to any network client | Unacceptable security risk |
| OAuth / OIDC | Industry standard, multi-user ready | Massive complexity for single-tenant local tool | Over-engineered |
| HTTP Basic Auth | Simple, browser-native | No WebSocket support, poor UX (modal prompt) | Does not cover WS |
| mTLS / client certificates | Very strong | Complex setup, poor DX for local dev | Too complex for target use case |
| Session cookie only (no token) | Simpler flow | Cannot protect standalone WS server on port 3100 | WS server has no cookie jar |

## Consequences

### Positive
- Terminal and HTTP endpoints are protected with minimal configuration
- Token auto-generated by default — zero-config for local development
- Constant-time comparison prevents timing attacks
- Cookie-based flow means the token is not visible in the URL after first redirect

### Negative
- Single shared secret — not suitable for multi-user deployments
- Token printed to stdout — must not run in shared terminal sessions
- Cookie + localStorage dual storage adds minor complexity

### Neutral
- `DEVDECK_TOKEN` env var provides a stable override for CI and scripting
- WebSocket close code 4401 is a custom code in the 4000-4999 private-use range
